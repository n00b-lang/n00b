# H3 module — overview

`include/net/quic/h3.h`, `include/net/quic/h3_types.h`, `src/net/quic/h3_*.c`.
Implements RFC 9114 (HTTP/3) on top of n00b's QUIC transport.  Phase
4 § 4.3 ships the **client** half + the shared frame layer; the
server half lands in 4.4.

HTTP/3 is a thin envelope on top of QUIC streams: each request is one
bidi stream; the body is a sequence of typed, length-prefixed frames.
Header compression is QPACK (RFC 9204), which n00b ships separately
in `include/net/quic/qpack.h`.

## What ships in v1 (sub-phase 4.3)

- **Frame encoder / decoder** (`h3_frame.c`).  All RFC 9114 § 7.2
  frame types: DATA (0x00), HEADERS (0x01), CANCEL_PUSH (0x03),
  SETTINGS (0x04), GOAWAY (0x07), MAX_PUSH_ID (0x0d).  Reserved
  frame types (0x02, 0x06, 0x08, 0x09 — RFC 9114 § 7.2.8) are hard-
  rejected; greased types are passed through (caller's
  responsibility to ignore).  Truncated input returns
  `N00B_QUIC_ERR_NEED_MORE_DATA`; the parser is stateless so callers
  feed bytes incrementally.
- **SETTINGS codec** (`h3_settings.c`).  Encoder + parser; rejects
  duplicate identifiers (RFC 9114 § 7.2.4) and reserved-from-HTTP/2
  identifiers (0x02–0x05 → SETTINGS_ERROR).  Unknown identifiers are
  ignored per § 7.2.4.1.
- **Client lifecycle** (`h3_client.c`).  Builds the client over an
  already-connected `n00b_quic_conn_t` (ALPN must have negotiated
  to `"h3"`); auto-opens the three uni streams (control + QPACK
  encoder + QPACK decoder), emits the initial SETTINGS frame, and
  exposes a request/response API.

The server half, server push (kind 0x01), trailers (TE: trailers
+ trailer HEADERS), and the streaming-iterator request API are
deferred to v1.1 (sub-phase 4.4) and 4.7.

## Public API surface

```c
/* One-shot frame primitives (server + client share these). */
n00b_result_t(size_t) n00b_h3_frame_emit_header(buf, type, length);
n00b_result_t(bool)   n00b_h3_frame_emit(buf, type, body, body_len);
n00b_result_t(bool)   n00b_h3_frame_parse(buf, offset, &out);
n00b_result_t(bool)   n00b_h3_frame_parse_bytes(data, data_len, &out);

/* SETTINGS body codec. */
n00b_result_t(bool) n00b_h3_settings_emit_body(buf, qpack_max, qpack_blocked, max_field);
n00b_result_t(bool) n00b_h3_settings_parse(body, body_len, &out);

/* Client. */
n00b_result_t(n00b_h3_client_t *) n00b_h3_client_new(conn, ...);
n00b_result_t(bool)               n00b_h3_client_drive(client);
n00b_h3_settings_t                n00b_h3_client_peer_settings(client);

/* Requests. */
n00b_result_t(n00b_h3_request_t *)  n00b_h3_client_request(client, method, scheme,
                                                            authority, path, ...);
n00b_result_t(n00b_h3_response_t *) n00b_h3_request_await(req, .deadline_ms = ...);
void                                n00b_h3_request_cancel(req);
uint64_t                            n00b_h3_request_stream_id(req);
```

## Wire format reference (RFC 9114 § 7)

```
Frame:
  varint(type) varint(length) payload[length]

Uni stream prelude (RFC 9114 § 6.2):
  varint(stream-kind)   then framed bytes per kind:
    0x00 Control       — SETTINGS + GOAWAY + MAX_PUSH_ID
    0x01 Push          — server-only (deferred)
    0x02 QPACK encoder — opaque QPACK encoder-stream bytes
    0x03 QPACK decoder — opaque QPACK decoder-stream bytes

Bidi request stream:
  HEADERS(qpack-encoded request fields)
  DATA(body)*
  [trailer HEADERS]    — deferred
```

## Hardening + DoS defenses

- **Frame body cap** — default 16 MiB
  (`N00B_H3_DEFAULT_MAX_FRAME_BODY`); configurable per client via
  the `.max_frame_body` kwarg on `n00b_h3_client_new`.  Larger
  frames produce `N00B_QUIC_ERR_FRAME_TOO_LARGE`.
- **Truncation** — every parser entry point handles short input
  cleanly via `N00B_QUIC_ERR_NEED_MORE_DATA`; nothing reads past
  the supplied length.
- **Reserved frame types** — RFC 9114 § 7.2.8: types 0x02, 0x06,
  0x08, 0x09 trigger an immediate connection error.  We surface
  this as `N00B_QUIC_ERR_PROTOCOL`; the caller closes the
  connection with `H3_FRAME_UNEXPECTED`.
- **SETTINGS duplicate identifier** — § 7.2.4: must be a
  connection error.  Same surface.
- **SETTINGS reserved-from-HTTP/2** — § 7.2.4.1: identifiers
  0x02/0x03/0x04/0x05 must trigger SETTINGS_ERROR.  Same surface.
- **Grease frames + grease settings** — passed through as ordinary
  frames/settings; the per-stream dispatcher silently drops them.

## Memory + threading discipline

- All long-lived objects (client, request, recv buffers, decoded
  header copies, response body buffers) come from
  `runtime->conduit_pool`.  `n00b_h3_alloc()` returns the pool
  pointer.
- One pthread mutex per client, one per request.  The IO thread
  (calling `n00b_h3_client_drive`) and the application thread
  (calling `n00b_h3_request_await`) interleave through the
  request's mutex; the request's `recv_buf` is only mutated by
  drive(), and only read once `state == DONE`.
- The per-uni-stream feeder scratch buffers (one per peer
  CONTROL / QPACK enc / QPACK dec stream) live on the client struct
  itself, so multiple H3 clients in the same thread are isolated.
  The earlier `__thread`-locals corner-cut was retired in 4.4's
  parallel-with-server pass.

## Server (sub-phase 4.4)

The server half lives in `src/net/quic/h3_server.c` and shares the frame
encoder/decoder + SETTINGS codec + uni-stream prelude with the client.

### Lifecycle

```c
n00b_h3_server_t *server = n00b_result_get(
    n00b_h3_server_new(listen_endpoint, conduit));

/* Subscribe to inbound requests. */
n00b_h3_request_inbox_t *inbox = n00b_h3_request_inbox_new(conduit);
n00b_h3_request_subscribe(n00b_h3_server_request_topic(server),
                          inbox, .operations = N00B_CONDUIT_OP_ALL);

/* IO loop — pump UDP, then drive the H3 server, then dispatch. */
while (running) {
    n00b_quic_endpoint_run_once(listen_endpoint, 5);
    n00b_h3_server_drive(server);

    while (n00b_h3_request_inbox_has_messages(inbox)) {
        n00b_h3_request_msg_t *m = n00b_h3_request_inbox_pop(inbox);
        n00b_h3_inbound_request_t *req = m->payload.req;
        /* …handle req, emit response… */
    }
}

n00b_h3_server_close(server);
```

`n00b_h3_server_new` requires the endpoint to have been built with
`.listen = true` and `.alpn = N00B_H3_ALPN`.  It subscribes to the
endpoint's accept topic; on each accepted QUIC connection the next
`n00b_h3_server_drive` call wraps the conn in a per-conn struct,
opens the three uni streams (CONTROL + QPACK enc + QPACK dec), and
emits an initial SETTINGS frame.  Inbound peer uni streams are
classified by their leading kind varint as they arrive; inbound bidi
(client-initiated, stream id mod 4 == 0) streams become
`n00b_h3_inbound_request_t`s.

Once a request's HEADERS have been QPACK-decoded AND the client has
FIN'd its half (so the body is complete), the server publishes a
`n00b_h3_request_event_t` on the request topic.  The application
dequeues the event from its inbox, inspects the pseudo-headers + body
via the accessors, and emits a response:

```c
const char *path = n00b_h3_inbound_request_path(req);
n00b_buffer_t *body = n00b_h3_inbound_request_body(req);

n00b_h3_inbound_request_respond(req, 200, NULL, 0,
                                (uint8_t *)"ok\n", 3);
```

`n00b_h3_inbound_request_respond` builds HEADERS (`:status` + extras)
followed by an optional DATA frame, FINing the stream by default.
`n00b_h3_inbound_request_reset(req, app_err)` resets the stream
without responding (e.g. for malformed pseudo-headers).

### GOAWAY on close

`n00b_h3_server_close(server)` initiates an orderly shutdown per
RFC 9114 § 5.2:

- For each per-conn control stream, emit a GOAWAY frame whose body is
  a single varint = `(highest-seen-client-bidi-stream-id) + 4`.  The
  `+4` is "the next id the client could have used" (client-initiated
  bidi ids are `0, 4, 8, …`).
- If no client request was ever issued on a given conn, the limit is
  `0`, refusing all future requests on that connection.
- Subsequent `n00b_h3_server_drive` calls are no-ops for new requests;
  in-flight ones already past the limit are still allowed to complete
  per § 5.2.
- The server's owned uni streams are closed; the QPACK encoder +
  decoder are released.

The matching client-side observer is in `process_control_frame`
(`h3_client.c`): a GOAWAY arriving on the peer CONTROL stream sets
`client->goaway_received` and ratchets `goaway_max_stream_id`
downward only.  New `n00b_h3_client_request` calls whose stream id is
≥ that limit fail synchronously with `N00B_QUIC_ERR_PEER_CLOSED`.

### Memory + threading discipline

- Per-conn + per-request structs come from the `n00b_h3_alloc()` pool
  (the conduit pool, same as the client).
- `n00b_h3_server_t` has its own pthread mutex; the IO thread and the
  request-handler thread interleave on it.  Each
  `n00b_h3_inbound_request_t` has its own mutex.
- The QPACK encoder + decoder are scoped to the server (one per
  server, not per conn) — RFC 9204 § 5 permits this when capacity is
  bounded; v1 starts at 4096 bytes per advertised setting and 0
  blocked streams.

### What's deferred past 4.4

- **Server push** — uni stream kind `0x01`, `PUSH_PROMISE` frame
  `0x05`.  The server STOP_SENDINGs an inbound push uni stream
  (RFC 9114 § 6.2.2 — only the server initiates push; receiving one
  is a protocol error).  Server-initiated push lands with sub-phase
  4.7 if/when the RPC layer needs it.
- **Trailers** — `TE: trailers` + a trailing HEADERS frame after the
  body.  Defer to v1.1.
- **Server-side streaming responses** — i.e. a response that emits
  multiple DATA frames over time.  v1's `_respond` is a one-shot:
  HEADERS + (optional DATA) + FIN.  A streaming variant is
  straightforward to add (a `_respond_begin` / `_respond_chunk` /
  `_respond_end` triple) and lands when 4.7's gRPC-shaped streaming
  RPCs need it.
- **Trust-store-based request authentication** — server-side
  validation of client mTLS certs and DPoP headers ties into the
  Phase 3 auth bridge; the H3 server hands the cert chain through to
  the auth layer when that bridge lands.

## Phase 4 hand-off

Sub-phase 4.4 (H3 server) builds on:
- The shared frame encoder/decoder (`h3_frame.c`).
- The shared SETTINGS codec (`h3_settings.c`).
- The same uni-stream prelude convention.

Sub-phase 4.6 (RPC unary) wraps `n00b_h3_client_request` with a
CBOR-bound stub:

```c
n00b_buffer_t *req_cbor = n00b_cbor_encode(...);
auto resp = n00b_h3_client_request(client, "POST", "https",
    "service.example.com", "/<svc>/<method>",
    .body = req_cbor->data, .body_len = req_cbor->byte_len,
    .extra_headers = &(n00b_h3_header_t){
        .name = (uint8_t *)"content-type", .name_len = 12,
        .value = (uint8_t *)"application/cbor+n00b", .value_len = 21,
    },
    .n_extra = 1);
auto reply = n00b_h3_request_await(resp);
n00b_cbor_decode(reply->body) → user struct.
```
